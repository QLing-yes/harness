### 自治理

#### 管理模板与实例
> 本实例是 Harness-Engineering 项目的自治理规范，基于 ./template/ 模板生成。
> **模板**（./template/）：含占位符，给用户克隆用。修改模板时必须同步更新本实例。
> **实例**（./harness/）：生成最佳实践规范，管理 harness 项目自身。
> 模板与实例之间固定内容保持严格一致，同时修改对应之一份，不得仅改一端。
> 模板与实例双向反哺：实例使用中暴露模板缺陷 → 改进模板 → 同步实例。

#### 版本管理
> 版本号三处同步：`INSTALL.md`（源头）→ `template/README.md` + `harness/README.md`
> 版本号格式：`> 版本: `vX.Y.Z``，位于文件最顶部第一行
> 当用户说"升级版本号"时，必须同步更新所有版本号。
> 版本号升级规则：X.Y.Z → X.Y.Z+1
> 自动升级版本号：当修改 template/ 下任何文件时，必须同步升级版本号。

### 铁律（绝对不可违反）

1. **模板与实例同步** — 修改 template/ 下任何文件时，必须同步更新 harness/ 下的同名文件
2. **链路不可跳过，禁止预读** — 严格按 → 下一步 逐级执行，不得跳过任何文件，不得自行决定读取顺序
3. **决策可追溯** — 非显而易见的决策必须写入 [决策记录](./DECISIONS.md)

### 自进化

用户说"分析"时做：

> 不乱进化，没有时不动 — 渐进式读取 → 节省 token → 提高命中率 → 用户效率和合理性才是终点。

> 维护 [项目结构](./README.md#项目结构) 列出的这些文件，随着以下分析和历史对话记录逐步建立清晰的项目模型和规范，不能为了维护而维护还不确定的就不改动。

1. 理解全貌 `[必须]` — 按链路读完项目，不跳步
2. 总结经验 `[必须]` — 回溯本次会话的坑、决策、修复
3. 外部学习 `[建议]` — 网络搜索最佳实践/工具/标准；优先官方文档、行业标准（Linux Foundation/AAIF）、高采纳度开源项目；拒绝 AI 生成内容农场；多源交叉验证后补充到项目（如无网络环境可跳过）
4. 检查优化项 `[建议]` — 以外部知识为参照，对照规范逐项审查遗漏和可改进点
5. 审视规范 `[按需]` — 基于优化项发现，审视规范是否需要移除、添加、修改、调整、完善
6. 自清洁 `[必须]` — 整理会话残留 + 执行一致性检查
   - 检查并提醒：
     - C1: 进行中任务 ≤ 3（超过则提醒移入[待办区](./tasks.md#待办区)）
     - C2: outputs/ 有产出文件但缺少 [INDEX.md](./outputs/INDEX.md) → 提醒创建索引
     - C3: 核心文件缺失（AGENTS / SESSION / specs / rules / tasks）→ 提醒补全
     - C4: SESSION.md 超过 3 天未更新 → 提醒补充最新进展
     - tasks.md 有已完成未归档任务 → 提醒归档
     - 临时备忘有内容 → 提醒处理
7. 自循环 `[必须]` — 再次验证"分析"结果，并将本次分析时间写入 [SESSION.md](./SESSION.md) 的「上次分析时间」

最终建议按必要性分级展示给用户：
- `[必须]` — 违反铁律、破坏框架完整性、或上下文必须延续
- `[建议]` — 提升质量或效率，非阻塞
- `[可选]` — 锦上添花

最终等待用户确认操作。
